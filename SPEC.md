# SPEC.md

## Goal

Build a clean combined plan-fact skeleton stage ETL and a canonical mart layer for plan-fact, YoY, MoM, IN-OUT, and planning-risk analysis.

## Requirements

- Read raw budget rows, DDS facts, p-fact control files, DDS article dictionary, and Cons Budget from `01_raw/`.
- Build p-fact valid triples by `Месяц + ЦФО + Код статьи ДДС`.
- Build p-fact valid pairs by `ЦФО + Код статьи ДДС`.
- Filter historical budget rows by valid triples and future budget rows by valid pairs.
- Mark player refunds as `Возврат` when article code is `CF4030400` and counterparty type is `Покупатель / Customer`.
- Exclude refunds from p-fact adjustment and reconciliation.
- Add `Корректировка p-fact` rows when union differs from p-fact by `>= 0.01`.
- Reconcile union to p-fact with tolerance `0.01`.
- Build `02_stage/01_full_stage.csv` only from adjusted `union_df` after `apply_p_fact_adjustments`.
- Aggregate adjusted rows to the analytical grain including date, article hierarchy, CFO, legal entity, counterparty, counterparty key, counterparty type, and currency.
- Normalize counterparty into `Контрагент` and `Ключ контрагента` without fuzzy matching or manual similarity rules.
- Use Cons Budget only for `IN` / `OUT` / `IN-OUT` service rows and monthly `IN-OUT, EUR`.
- Mart generation uses `02_stage/01_full_stage.csv` as its stage source and writes production marts to `03_marts/`, signals to `04_signals/`, evidence to `05_evidence/`, reports to `06_reports/`, and QA to `07_qa/`.
- Do not include `Тип операции`, `Источник данных`, `Направление`, `Статья 3`, `План - Факт, EUR`, `ABS отклонения, EUR`, `% исполнения бюджета`, `Направление отклонения`, or `Доля от IN-OUT, %`.

## 01_full_stage.csv QA / Data Contract Notes

- Read the CSV with `sep=';'`, `decimal=','`, and `encoding='utf-8-sig'`.
- `Дата` is stored as text in the CSV. Parse it explicitly before time-series analysis.
- Treat `Ключ контрагента`, `source_row_id`, and `Код статьи ДДС` as string/key fields, not numeric measures.
- `source_row_id` is grouped lineage after stage aggregation and may contain multiple raw row ids separated by ` | `.
- `source_mix = cons_budget` identifies Cons Budget `IN` / `OUT` / `IN-OUT` service rows. These rows are not part of detailed expense analysis and are not part of p-fact reconciliation.
- `IN-OUT, EUR` is a monthly window-like value repeated for every row in the same month. Do not sum `IN-OUT, EUR` across rows.
- Main base skeleton financial measures are `План, EUR`, `Факт, EUR`, and `IN-OUT, EUR`; these are allowed in this project's stage contract.
- `Сумма исходная` is trace/diagnostic only and must not be used for reconciliation.
- Derived business metrics, risk labels, materiality flags, management conclusions, and recommendations are forbidden in stage.
- `load_timestamp` / `pipeline_run_id` is not currently present and is a known limitation / future additive improvement.
- A dedicated `rejected_rows` / validation errors artifact is not currently present and is a known limitation / future additive improvement.
- Period parsed from filename is a candidate period unless independently confirmed by source metadata or user instruction.
- Any code change for these limitations requires separate approval and must be backward-compatible.

## Mart Contract

Mart outputs are generated by:

```bash
python3 src/build_marts.py
```

Required production mart files:

- `03_marts/mart_10_stage_typed.parquet`
- `03_marts/mart_11_month_inout.parquet`
- `03_marts/mart_12_article_month.parquet`
- `03_marts/mart_13_article_cfo_month.parquet`
- `03_marts/mart_14_article_cfo_counterparty_month.parquet`
- `03_marts/mart_15_article_year.parquet`
- `03_marts/mart_16_article_cfo_year.parquet`
- `03_marts/mart_17_article_cfo_counterparty_year.parquet`

`03_marts/` must contain only production mart tables. Signal, evidence, QA, reports, memo text, recommendations, and LLM fields must live outside the mart layer.

Required signal files:

- `04_signals/signal_01_scale.parquet`
- `04_signals/signal_02_yoy.parquet`
- `04_signals/signal_03_mom.parquet`
- `04_signals/signal_04_localization.parquet`
- `04_signals/signal_05_plan_risk.parquet`
- `04_signals/signal_06_counterparty_plan_risk.parquet`
- `04_signals/signal_07_inout_weight.parquet`

Required evidence files:

- `05_evidence/evidence_cards.json`
- `05_evidence/evidence_cards.csv`
- `05_evidence/llm_context_package.json`

Required report files:

- `06_reports/budget_mart_excel.xlsx`
- `06_reports/analytical_memo.md`

Required QA files:

- `07_qa/qa_report.json`
- `07_qa/qa_reconciliation.csv`
- `07_qa/qa_inout.csv`
- `07_qa/qa_counterparty_quality.csv`
- `07_qa/qa_forbidden_claims.csv`

Parquet production marts may use stable internal snake_case columns. User-facing CSV/XLSX exports must use Russian business-readable columns.

Financial formulas:

- `plan_fact_eur = plan_eur - fact_eur`
- `abs_deviation_eur = abs(plan_fact_eur)`
- `execution_pct = fact_eur / plan_eur`; null when denominator is zero.
- `in_out_eur` is month-level and must not be summed across rows.
- `effective_period_type` is required and is used for mart calculations.

## Risks

- Raw workbook format changes can break parsing.
- p-fact and raw DDS can intentionally differ because refunds are excluded from reconciliation but remain visible in the skeleton stage.
